# 하둡 및 스파크

생성일: 2026년 2월 24일 오전 12:40

## **1. 자료 설명**

---

## 1) 전체 흐름 한 줄 요약

**API로 원본(JSON) 수집 → HDFS에 “원본 그대로” 저장 → Spark가 HDFS에서 읽어 정제/집계 → HDFS에 “정제 결과(Parquet)” 저장 → 이후 서비스/분석은 clean을 사용**

---

## 2) Hadoop(HDFS)이 하는 일: “분산 저장소”

### 역할

- HDFS는 **파일을 안전하게, 분산해서 저장**하는 스토리지.
- 파일을 올리면(NameNode에게 “메타데이터” 등록) → 실제 데이터는 DataNode들에 저장됨.

### 구성

- **NameNode**
    - “어떤 파일이 어떤 블록들로 나뉘었고, 그 블록이 DataNode 어디에 있는지” 관리하는 **목차/관리자**
- **DataNode(여러 대)**
    - 실제 파일 내용(블록)을 들고 있는 **창고**

### “분산 저장”이 무슨 뜻이냐

- 너희가 `/data/raw/naver/...`에 파일을 넣으면
    - 파일이 HDFS 내부에서 **블록 단위**로 쪼개지고
    - 블록들이 여러 DataNode로 퍼져서 저장됨
    - 보통 **replication(복제)**도 걸려서 장애 나도 안 날아가게 함
        
        (실습에서는 설정에 따라 복제가 1이거나 3일 수 있음)
        

---

## 3) Spark가 하는 일: “분산 처리 엔진”

### 역할

- Spark는 **분산된 데이터를 여러 CPU/노드에서 나눠서 처리**하는 엔진.
- HDFS에 저장된 raw를 읽어서,
    - JSON 파싱
    - 필요한 컬럼 뽑기
    - 정제(가격 숫자화, 필터링)
    - 집계(중앙값/최소/최대 등)
    - 결과를 Parquet로 저장

### 구성

- **Driver(스파크 잡의 두뇌)**
    - “작업을 어떤 단계로 나눌지” 계획하고, Executor들에게 일을 배분
- **Executor(실제 일꾼, 보통 worker에 떠있음)**
    - 데이터를 읽고 변환하고 집계하는 **실제 연산 수행**
- **Spark Master/Worker(Standalone 모드 기준)**
    - Master는 자원 배치(어느 worker에 executor를 띄울지)
    - Worker는 executor를 띄우는 서버

---

## 4) 너희가 한 파이프라인을 단계별로 “정확히” 말하면

### (A) Raw 수집 단계 (로컬/WSL)

- 네이버 쇼핑 API를 쿼리 200개로 호출
- 결과를 **원본 JSON 그대로** 파일로 저장
    
    예: `raw/naver/date=2026-02-23/query_*.json`
    

> 이 단계는 “분산”이 아니라 그냥 수집 단계. 중요한 건 “원본을 훼손하지 않고 남긴다”는 것.
> 

---

### (B) Raw 적재 단계 (HDFS에 저장)

- 로컬 파일을 namenode 컨테이너로 `docker cp`
- `hdfs dfs -put`으로 HDFS에 업로드
    
    예: `/data/raw/naver/date=2026-02-23/`
    

이때 HDFS 내부에서는:

- NameNode가 “이 파일은 블록들이 있고, DataNode1/2/3에 이렇게 저장됨”을 기록
- DataNode들이 실제 블록을 들고 있음

---

### (C) Spark 정제 단계 (Spark Job)

Spark job이 하는 일(핵심 로직):

1. HDFS에서 `query_*.json`들을 읽음
2. JSON의 `items` 배열을 `explode`해서 “상품 단위 row”로 펼침
3. `lprice`를 숫자로 캐스팅하고 0/빈값 제거
4. `query_key` 기준으로 중앙값/최소/최대/개수 같은 통계 집계
5. 결과를 HDFS `/data/clean/...`에 **Parquet**로 저장

> 여기서 “분산 처리”는
> 
> 
> **200개 파일을 여러 파티션으로 나누고**, executor들이 동시에 읽고 처리하는 걸 말함.
> 

---

### (D) 결과 확인 단계

- HDFS에 Parquet가 생성됐는지 확인
- Spark로 다시 읽어서 `count()`, `show()`로 검증

---

## 5) “왜 raw와 clean을 나눠 저장하냐”를 이렇게 말하면 설득력 있음

- raw는 **재처리 가능한 원본(데이터 레이크의 원칙)**
    
    → 로직 바꾸거나 버그 수정하면 언제든 다시 clean 생성 가능
    
- clean은 **빠르게 쓰기 위한 정제 데이터(분석/서비스 친화 포맷)**
    
    → Parquet는 컬럼 기반이라 분석/집계가 빠름
    

---

## 6) 이번에 너희가 겪은 문제(권한)도 설명 포인트로 좋음

- Spark job이 HDFS에 쓸 때 user가 `spark`였고,
- HDFS `/data`는 `root:supergroup`만 쓰게 되어 있어서
- **write 단계에서 계속 실패** → `/data/clean`이 안 생김
    
    즉, “분산에서 자주 터지는 건 로직보다 권한/설정”이라는 좋은 교훈.
