# Jira Epic/Story 운영 가이드 (S14P21D206)

## 1. 문서 목적
`jira.md`와 `LLM_GUIDE.md`를 기준으로, 우리 팀이 Jira에서 Epic/Story를 어떤 방식으로 작성/운영할지 통일한다.

## 2. 핵심 원칙
- Epic: `{도메인명}` 단위(관련 기능 묶음)
- Story: 사용자 목적 중심 문장 + 구현 기능 단위
- Task: 회의/문서/운영성 업무 등 코드 구현 외 작업
- Bug: QA/운영 중 발견된 결함

## 3. 이슈 타입 사용 기준
- `Epic`: 기능군의 상위 묶음
- `Story`: 사용자 가치가 있는 기능 단위
- `Task`: 미팅, 문서화, 운영 작업
- `Bug`: 오류 수정

## 4. 제목 규칙 (jira.md 기준)
아래 Prefix를 제목 맨 앞에 사용:
- `Meeting:` 회의
- `Docs:` 문서 작업
- `Feat:` 기능 구현
- `Fix:` 버그 수정
- `Refactor:` 코드 개선
- `Style:` UI/Figma 포함 스타일 작업
- `Rename:` 파일/폴더명 변경
- `Remove:` 파일/기능 삭제

권장 예시:
- `Feat: 예산/기간/위험민감도 입력 API 구현`
- `Style: 메인 추천 리스트 카드 UI 구현`
- `Fix: 뉴스 탭 빈 응답 시 크래시 수정`

## 5. Story 작성 템플릿
Story 설명은 아래 형태를 기본으로 사용:

```md
[Role] 사용자는 {목적}을 하기 위해 {기능명}을 사용하고 싶다.

## ✅ 완료 조건
- 

## 💬 기타
- 추가 공유사항/이슈
```

Role 값은 파트 기준으로 명시:
- `FE`, `BE`, `INF`, `BD` (필요 시 `AI`를 팀 합의로 추가)

## 6. 스토리 포인트 기준 (jira.md 기준)
- 1pt: 매우 단순, 리스크 거의 없음
- 3pt: 기준 스토리(보통 난이도)
- 5pt: 복잡 로직/통합 필요
- 8pt: 고복잡/불확실/협업 다수
- 13pt: 매우 큼(가능하면 분할 권장)

운영 기준:
- 1pt = 1시간
- 1인 1일 8pt, 1주 40pt 기준으로 분배
- 진행중 이슈는 실시간 기준 1~2개 유지

## 7. 우리 프로젝트 Epic 분해안 (기능명세 기준)

### EPIC-AUTH: 인증/사용자/선호
대상 PAGE: 101, 102, 203, 301
- Google 로그인
- 유튜브 연동 동의 상태
- 선호 수정/회원탈퇴(소프트 딜리트)

### EPIC-RECO: 추천 메인/근거
대상 PAGE: 201, 202, 206
- 예산/기간/위험민감도 입력
- 추천 리스트 + 지구본 시각화
- 추천 이유(근거) 표시

### EPIC-DETAIL: 상세 모달/저장
대상 PAGE: 205, 205-1
- 지역 상세 모달
- 조회 시점 데이터 저장(리포트 원본)

### EPIC-DATA-TABS: 물가/항공/뉴스
대상 PAGE: 207, 207-1, 208, 210, 302
- 물가 비교/검색
- 항공권 정보 탭
- 뉴스 요약/목록

### EPIC-INFRA: 배포/관측/안정화
대상: 공통
- CI/CD
- 모니터링/알람
- fallback/장애 대응

### EPIC-DEMO: 발표 준비
대상: 4주차 목~금
- 데모 시나리오 고정
- 백업 플로우/영상 준비
- Q&A 문서화

## 8. Epic-Story-Task 연결 방식
- Story는 반드시 하나의 Epic에 연결
- Story 하위의 구현 단위는 필요 시 Task로 쪼갬
- Bug는 발생 Epic에 연결하여 추적

권장 구조:
1. Epic 생성
2. Story 생성 (`Feat:` 중심)
3. Story를 Epic에 링크
4. 문서/회의는 `Docs:`/`Meeting:` Task로 분리
5. QA 중 결함은 `Fix:` Bug 발행

## 9. 4주 일정 반영 우선순위
- 4주차 수요일까지 완료 대상 Epic:
  - `EPIC-AUTH`
  - `EPIC-RECO`
  - `EPIC-DETAIL`
  - `EPIC-DATA-TABS` (PAGE-401 제외)
  - `EPIC-INFRA`(최소 배포/모니터링)
- 4주차 목~금:
  - `EPIC-DEMO` 중심 운영

## 10. LLM_GUIDE 기반 Jira CLI 운영 규칙

### 읽기(조회) 작업
반드시 `search_issues.py` 사용:
1. `python search_issues.py ...` 실행
2. `TEMP_FILE_PATH` 확인
3. temp markdown 파일 읽기
4. 결과 공유

### 생성/수정 작업
- 생성: `create_issues.py`
- 수정: `update_issues.py`
- Epic 연결: `link_epic.py`

주의:
- Windows 인코딩 이슈 때문에 터미널 출력 직접 파싱 금지
- 기본 프로젝트 키는 `S14P11D206`로 되어 있으니 현재 프로젝트 키와 다르면 명시적으로 지정

## 11. 바로 쓸 수 있는 Story 예시

### 예시 1 (FE)
- 제목: `Feat: 메인 페이지 예산/기간/위험민감도 입력 폼 구현`
- 설명:
```md
[Role] 사용자는 본인 예산과 기간에 맞는 여행지를 추천받기 위해 입력 폼을 사용하고 싶다.

## ✅ 완료 조건
- 예산/기간/위험민감도 입력 가능
- 유효성 검사 및 에러 메시지 표시
- 제출 시 추천 API 호출

## 💬 기타
- PAGE-201 기준
```

### 예시 2 (BE)
- 제목: `Feat: 추천 결과 API(리스트+근거) 구현`
- 설명:
```md
[Role] 사용자는 추천 결과의 이유를 함께 확인하기 위해 추천 결과 API를 사용하고 싶다.

## ✅ 완료 조건
- Top-N 도시 반환
- 각 도시별 근거 항목(물가/위험/취향/항공) 포함
- 응답 실패 시 fallback 응답

## 💬 기타
- PAGE-201, 206 연동
```

### 예시 3 (AI/LLM)
- 제목: `Feat: 추천 이유 생성 프롬프트 v1 적용`
- 설명:
```md
[Role] 사용자는 추천 결과를 신뢰하기 위해 추천 이유 설명 기능을 사용하고 싶다.

## ✅ 완료 조건
- 입력 스키마(예산/기간/위험/취향) 반영
- 출력 스키마(JSON) 준수
- 형식 오류 시 재시도 또는 기본 문구 반환

## 💬 기타
- 모델 학습 없이 프롬프팅 기반
```

## 12. 합의 필요 항목
